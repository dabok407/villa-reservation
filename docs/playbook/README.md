# AI 주도 개발 플레이북 (AI-Driven SDLC Playbook)

> **목적**: 아무것도 없는 환경에 **Claude(AI 도구) 하나만** 있어도, 작업 유형에 맞게
> 처음부터 끝까지(A-Z) 일관된 기준으로 개발·검증·문서화를 수행하기 위한 절차 묶음.
> 라이브코딩 면접에서도 이 순서 그대로 운전하면 "과정과 판단"이 그대로 드러난다.
>
> **구성**: 이 문서(전체 흐름) + [gates.md](gates.md)(결정 게이트 3종) + [../dev-checklist.md](../dev-checklist.md)(스펙 심문) + [deliverables.md](deliverables.md)(산출물 카탈로그).
> 실제 적용 사례: [../ARCHITECTURE.md](../ARCHITECTURE.md), [../verification-log.md](../verification-log.md).

## 0. 작업 유형부터 고른다 (어떤 페이즈를 도는가)

| 작업 유형 | 도는 페이즈 | 비고 |
|-----------|-------------|------|
| **신규 구축** | P0→P1→P2→P3→P4→P5→P6 (전체) | 요구 수집부터 |
| **유지보수(버그픽스)** | P0(재현)→P4(실패테스트)→P3(수정)→P4(회귀)→P6(델타) | 먼저 깨지는 테스트로 고정 |
| **개선(기능추가)** | P0(영향분석)→P1(미니범위)→P2(델타설계)→P3→P4→P6(델타) | 기존 불변식 보존 우선 |

> 유형을 정하면 불필요한 페이즈를 건너뛴다. 단, **건너뛴 페이즈는 "왜 생략했는지" 한 줄 기록**.

## 1. 7개 페이즈 (A→Z)

각 페이즈는 **목적 / 활동 / 산출물 / 게이트**로 구성. "이 프로젝트에서 한 것"은 별장 사전과제 실제 사례.

### P0 — 파악 (Discover)
- **목적**: 현황·요구를 사실로 확정 (추정 금지)
- **활동**: (기존) 코드 **읽기만** 분석 / (신규) 요구사항 인터뷰 → 기술스택·도메인·핵심경로·제약 정리
- **산출물**: 현황 분석 보고 / 요구사항 정의서
- **이 프로젝트**: STEP 0 — 별장 코드 분석 보고 (기술스택, 도메인모델, 핵심경로, 삽입지점)

### P1 — 범위·계획 (Scope) → 🚦 게이트 1
- **목적**: MVP 경계 확정, 욕심 차단
- **활동**: 한 문장 정의 / 포함·제외 / 완료 정의 / 일정·리스크
- **산출물**: 범위 정의서 (포함 + **명시적 제외 목록**)
- **이 프로젝트**: "MVP=조회·검증·응답까지, 쓰기/RAG/추천 제외"

### P2 — 설계 (Design) → 🚦 게이트 2
- **목적**: 구조·불변식·인터페이스를 코드 짜기 전에 확정
- **활동**: [dev-checklist.md](../dev-checklist.md) 11개 카테고리 **심문** → 아키텍처·DB·API 설계, 불변식 식별, 최소침습 삽입 지점, 기존 재사용 vs 신규
- **산출물**: 아키텍처/흐름 설계서, DB·API 설계, 테스트 전략
- **이 프로젝트**: STEP 3 — LLM 연계+tool use 설계, 충돌쿼리 재사용, 가구매핑 프롬프트 주입

### P3 — 구현 (Implement)
- **목적**: 작은 슬라이스로 쪼개 구현
- **활동**: 슬라이스 단위(검증 가능한 최소). 하네스 규칙(CLAUDE.md) 준수. AI 출력은 **읽고 판단·되돌림**
- **산출물**: 동작 코드 (+ 슬라이스마다 테스트)
- **이 프로젝트**: STEP 4 슬라이스 1 — check_availability 도구 (LLM 없이 검증되는 것부터)

### P4 — 검증 (Verify) → 🚦 게이트 3
- **목적**: 도메인 규칙 기준으로 출력을 의심하고 깨본다
- **활동**: 엣지케이스 목록 → 단위/통합/E2E. 버그는 **먼저 실패 테스트로 고정** 후 수정
- **산출물**: 검증 로그 (입력/기대/실제/판정/조치)
- **이 프로젝트**: 5케이스 단위 테스트 PASS → [verification-log.md](../verification-log.md)

### P5 — 배포·운영 (Deploy/Operate)
- **목적**: 실제 환경에 안전하게 올리고 흔적을 남긴다
- **활동**: 리소스 점검 → 조치(스왑·힙튜닝) → 빌드 → 배포 → 헬스체크. 키는 환경변수
- **산출물**: 운영 가이드, 배포·점검 흔적 로그
- **이 프로젝트**: EC2 점검(410Mi/swap0) → 스왑 계획 → [deploy-ec2 스킬](../../.claude/skills/deploy-ec2/SKILL.md)

### P6 — 문서·인수인계 (Document)
- **목적**: 비개발자/개발자 모두 이해 + 라인 단위로 안 읽어도 구조 장악
- **활동**: [deliverables.md](deliverables.md)에서 작업 유형에 맞는 산출물 선택·작성
- **산출물**: README, 시스템 개요, 흐름 매핑, 운영/사용자 가이드 등
- **이 프로젝트**: README(AX 프레이밍 + 검증 깊이), ARCHITECTURE.md

## 2. 블랭크 환경 부트스트랩 (첫 10분에 할 것)

Claude만 있는 생환경에서 시작할 때 순서:
1. **파악 먼저** — "코드 수정하지 말고 읽기만 해서 기술스택·구조·핵심경로 보고해" (P0)
2. **CLAUDE.md 생성** — `/init` 후 다듬기. 프로젝트 규칙 + "지켜야 할 불변식" 박기
3. **가드 훅(선택)** — 시크릿 평문·실데이터 보호가 필요하면 PreToolUse 훅
4. **게이트 1로 범위 합의** — [gates.md](gates.md)의 범위 게이트를 채운다
5. 이후 P2→P6 진행

> 면접에서 환경이 빈약하면: "먼저 파악하고, 규칙을 CLAUDE.md에 박고, 작게 쪼개 검증하며 갑니다"를 **선언**하고 시작 — 방법론 자체가 평가 포인트.

## 3. 라이브코딩 적용 (말로 하는 게이트)

준비 단계의 "사람에게 점검받기"는, 면접에선 "**스스로 판단하고 소리내어 중계**"로 바뀐다.
- 각 🚦 게이트에서 멈춰 [gates.md](gates.md) 항목을 *말로* 짚는다.
- AI 출력마다: "이건 맞다(이유) / 이건 위험(이유) → 되돌림". **되돌리는 장면이 최고 가점.**
- 검증은 눈앞에서 실행해 결과를 짚는다. "되겠지" 금지.
