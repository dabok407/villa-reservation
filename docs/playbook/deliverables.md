# 산출물 카탈로그 (Deliverables Catalog) — 표준 문서 + 미니 템플릿

> 신규 시스템을 만들면 "보통 나와야 할" 산출물들을 정형화. 작업 유형별로 **무엇이 필수/선택인지**
> 고르고, AI에게 "코드 분석해서 이 형식으로 채워줘" → 사람이 실제 코드와 대조(spot-check).
> 핵심: 전부 라인 단위로 읽지 않고 **불변식·경계·핵심 흐름만** 문서로 장악한다.

---

## 1. 산출물 선택표 (작업 유형별)

| 산출물 | 목적 | 신규 | 유지보수 | 개선 |
|--------|------|:---:|:---:|:---:|
| **프로젝트 설명서**(시스템 개요·가치·대상) | 무엇을·왜 | 필수 | – | 델타 |
| **요구사항 정의서** | 범위·기능 명세 | 필수 | – | 미니 |
| **아키텍처 문서**(SW/HW/네트워크) | 구성·배치·통신 | 필수 | – | 영향분석 |
| **DB 설계서**(ERD·테이블 명세) | 데이터 구조 | 필수 | 변경분 | 변경분 |
| **API 명세서** | 인터페이스 계약 | 필수 | 변경분 | 변경분 |
| **프로세스 흐름도 / 기능 흐름 매핑** | 화면→API→코드→DB | 필수 | – | 델타 |
| **기능 설계도** | 기능별 상세 로직 | 핵심기능 | – | 신규기능 |
| **테스트·검증 로그** | 무엇을 어떻게 확인 | 필수 | **필수** | 필수 |
| **운영 가이드**(배포·모니터링·롤백) | 운영자용 | 필수 | 변경분 | – |
| **사용자 가이드** | 사용자용 | 필수 | – | 신규기능 |
| **결정 로그(ADR-lite)** | 왜 이렇게 정했나 | 권장 | 권장 | 권장 |
| **README** | 진입점·요약 | 필수 | 갱신 | 갱신 |

> "–"는 보통 불필요. 단 생략 시 한 줄 사유 기록.

## 2. 미니 템플릿 (각 산출물 뼈대)

### 프로젝트 설명서
```
# <시스템명>
- 한 줄 정의 / 핵심 가치 / 타겟 사용자
- 주요 기능 (목록)
- 기술 스택 / 아키텍처 개요(요약 다이어그램)
- 비기능 요구(보안·성능·가용성)
```

### 요구사항 정의서
```
- 배경/목적
- 기능 요구(우선순위 표: 필수/선택)
- 비기능 요구
- 범위 제외(scope guard)
- 가정·제약
```

### 아키텍처 문서 (SW/HW/네트워크)
```
- 소프트웨어: 계층/모듈/의존 방향, 외부연동
- 하드웨어/인프라: 서버 스펙, 리소스 한계, 배치
- 네트워크: 토폴로지(클라→프록시→앱→DB), 포트, HTTPS/방화벽
- 배포 단위(JAR/컨테이너), 환경 분리(로컬/운영)
```

### DB 설계서
```
- ERD (텍스트 다이어그램 가능)
- 테이블별: 컬럼/타입/제약/인덱스/관계
- 마이그레이션 이력
- 데이터 흐름(생성→변경→소멸)
```

### API 명세서
```
| 메서드 | 경로 | 설명 | 요청(body/param) | 응답 | 상태코드/에러 |
- 인증 방식 / 공통 에러 포맷
```

### 프로세스 흐름도 / 기능 흐름 매핑  ← 실제 예시: ../ARCHITECTURE.md
```
| 화면/트리거 | HTTP | Controller | Service | Repository/쿼리 | 테이블 | 핵심 규칙 |
+ 토폴로지 다이어그램 + 도메인 불변식 목록
```

### 기능 설계도 (기능별)
```
- 입력/출력/전제조건
- 처리 로직(단계별), 분기/예외
- 의존(다른 기능/외부 API)
- 엣지케이스
```

### 테스트·검증 로그  ← 실제 예시: ../verification-log.md
```
### [케이스] 입력 / 기대(도메인 규칙) / 실제 / 판정 PASS·FAIL / 조치
```

### 운영 가이드
```
- 빌드·배포 절차 / 롤백
- 환경 설정(환경변수·프로필)
- 헬스체크 / 로그 위치 / 모니터링
- 리소스 점검·튜닝(메모리/힙)
- 장애 대응(OOM, 타임아웃)
```

### 사용자 가이드
```
- 주요 화면별 사용법(스크린샷)
- 자주 묻는 작업 / FAQ
```

### 결정 로그 (ADR-lite)
```
### [결정] 무엇을 / 배경 / 선택지 / 택한 것 / 이유 / 트레이드오프
예: "RAG 대신 LLM API + function call — 임베딩 미보유, 정직성, 우대사항 충족"
```

## 3. 운용 원칙
1. **생성은 AI, 검증은 사람** — AI에게 채우게 하고, 실제 코드와 대조해 거짓/누락을 잡는다.
2. **불변식·경계 우선** — 모든 라인이 아니라, 흐름 매핑의 "핵심 규칙" 열과 불변식만 집중 검증.
3. **살아있는 문서** — 코드 바뀌면 문서도 델타만 갱신. 결정 로그로 "왜"를 남긴다.
4. **봇 적용** — 코인봇/주식봇도 이 표에서 필수 산출물만 1장씩 뽑으면, 라인 단위 정독 없이 구조를 인수인계·설명할 수 있다.
