# AI 협업 개발 — 스펙 심문 체크리스트 (범용 템플릿)

> **용도**: AI 도구로 무언가를 만들기 *전에*, 이 항목들을 빠르게 훑어 "이 기능에 해당되나?"를 묻는다.
> 해당되면 설계에 반영하고, **해당 안 되면 "불필요"라고 명시적으로 기록**한다(나중에 "왜 안 했나" 방어용).
> 핵심 메시지: *"나는 내 나름의 기준이 있어서, 만들기 전에 이걸 다 점검하고 들어간다."*
> 프로젝트 무관 재사용. 필요 시 `.claude/skills/spec-check`로 승격해 `/spec-check`로 호출 가능.

---

## 0. 요구사항·범위 (가장 먼저)
- [ ] 무엇을/왜 만드는지 **한 문장**으로 말할 수 있나
- [ ] MVP 경계는? **이번에 안 하는 것**을 명시했나 (scope guard)
- [ ] 완료 정의(Definition of Done)가 측정 가능한가

## 1. 도메인 규칙·불변식
- [ ] 절대 깨지면 안 되는 규칙은? (예: 예약 충돌 = 반개구간)
- [ ] 그 규칙의 **단일 진실 원천**은 어디인가 — 중복 구현 위험은 없나
- [ ] 경계 조건: off-by-one, 동일값, 빈 값/null, 음수, 시간대(UTC/KST)

## 2. 데이터 · DB
- [ ] 엔티티/테이블 스키마, 키, 인덱스 (쿼리 패턴에 맞나)
- [ ] 트랜잭션 경계 — 어디까지 원자적이어야 하나 / 읽기는 readOnly인가
- [ ] **의도치 않은 쓰기·사이드이펙트** 차단했나 (조회 기능이 데이터 변형 안 하나)
- [ ] 마이그레이션 전략 (스키마 변경 이력 관리)
- [ ] **실데이터 안전** — 운영 DB 덮어쓰기/삭제 방지
- [ ] 동시성 — 락(낙관/비관), 멱등성

## 3. API · 인터페이스 형태 (★ 자주 놓치는 곳)
- [ ] 동기 REST로 충분한가
- [ ] **배치/스케줄러** 필요? (대량 처리, 주기 작업, 정산, 집계)
- [ ] **실시간 양방향** 필요? (WebSocket/SSE — 채팅, 실시간 시세, 푸시 알림)
- [ ] **비동기 큐(MQ)** 필요? (느린 작업 분리, 부하 분산, 재시도, 이벤트)
- [ ] 멱등성 (중복 요청/재전송 안전)
- [ ] 페이지네이션·필터·정렬 / 버저닝·하위호환

## 4. 보안
- [ ] 인증/인가 — 누가 무엇에 접근 가능한가
- [ ] 비밀정보(키·비번) — 코드/설정/로그에 평문 금지, 환경변수
- [ ] 입력 검증 — SQL injection(파라미터 바인딩), 타입·범위
- [ ] XSS / CSRF (웹)
- [ ] 민감정보(PII) 노출·로깅 여부
- [ ] 권한 상승 / IDOR (남의 리소스 id로 접근)
- [ ] **외부 입력 신뢰 경계** — LLM 출력·사용자 입력은 신뢰하지 않는다

## 5. 프론트 · 화면
- [ ] 화면 복잡도 → **라이브러리 선택 근거** (바닐라로 될 걸 무겁게/그 반대도 금지)
- [ ] 상태관리 범위 — 로컬 state로 충분한가, 전역(store) 필요한가
- [ ] 반응형 · 모바일 · 다크모드
- [ ] 접근성 — 키보드, 포커스 트랩, ESC, 스크린리더
- [ ] 로딩 / 에러 / 빈 상태(empty) 처리
- [ ] 기존 디자인 시스템·컴포넌트 일관성

## 6. 성능 · 확장
- [ ] 병목 예측 — N+1, 풀스캔, 직렬 외부호출
- [ ] 캐시 필요성·무효화(TTL/이벤트) 전략
- [ ] **리소스 한계** — 메모리/CPU (예: EC2 가용 410Mi)
- [ ] 부하 시 동작 — 타임아웃, 백프레셔, 레이트리밋

## 7. 운영 · 배포
- [ ] 환경 분리 (로컬/스테이징/운영, 프로필)
- [ ] 설정 외부화 (환경변수/EnvironmentFile)
- [ ] 로깅 · 모니터링 · 알림
- [ ] 배포 절차 · **롤백 가능성** · 헬스체크
- [ ] 리버스 프록시 · HTTPS

## 8. 실패 모드 · 복원력
- [ ] 외부 의존(API/DB) 실패 시 — 타임아웃, 재시도(백오프), 회로차단
- [ ] 부분 실패 · 재시작 안전(graceful), 미완료 작업 복구
- [ ] 사용자에게 보일 **fallback** — 500/스택트레이스 노출 금지

## 9. 검증 · 테스트
- [ ] 단위(핵심 로직·불변식) / 통합 / E2E — 어디까지 할지
- [ ] **엣지케이스 목록**을 먼저 적었나
- [ ] 회귀 방지 — 버그는 **먼저 실패하는 테스트로 고정** 후 수정
- [ ] 검증 흔적 기록 (로그/문서) — "무엇을 어떻게 확인했나"

## 10. AI 특화 (LLM 연계 시)
- [ ] 환각 차단 — 도구 결과/출처 **인용 강제**, 모르면 "모름"
- [ ] 비용·지연 — 토큰 한도, tool 루프 상한, 프롬프트 캐싱
- [ ] 프롬프트 주입 방어 (사용자 입력이 시스템 지시를 덮지 못하게)
- [ ] 결정성·재현성 — 같은 입력에 일관된 출력인가
- [ ] **모델 하드코딩 금지** — 설정값으로, 교체 가능하게

---

### 쓰는 법 (3분 루틴)
1. 새 기능 시작 전 0~10번을 훑는다.
2. 해당되는 항목만 골라 설계 메모에 적는다.
3. **해당 안 되는 항목도 "불필요"로 한 줄 남긴다** → "검토했고 의도적으로 제외"의 증거.
4. 라이브코딩에선 이 중 2~3개를 소리내어 짚는다: *"이건 읽기 전용이라 사이드이펙트 0, 키는 환경변수, 경계값은 동일일 케이스를 테스트로 고정합니다."*
